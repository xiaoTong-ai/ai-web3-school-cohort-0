### 阅读：Context

### 关键概念 / 收获

1. **Context 的本质** — 模型这一次能看见、能使用、能被影响的信息空间。Context 不是简单的长文本拼接，而是一套信息治理问题。

2. **Context Window** — 窗口越大 ≠ 越聪明。关键信息要直接给，长文档按需取，低可信内容隔离标注。模型"看见了但没抓住重点"是长上下文常见问题。

3. **Context Engineering** — 设计上下文进入模型的方式，包括选择数据源、排序、裁剪、标注来源、隔离不可信内容。目标不是塞满窗口，而是让模型在正确的信息层级里工作。

4. **Memory** — 跨请求保留的信息（偏好、历史等），方便但有隐性风险。不能替代实时授权——用户过去允许某个动作，不代表现在仍然允许。

5. **Knowledge Base** — 可检索的外部知识库，解决模型知识过期问题。需要维护来源、更新时间、版本，否则 RAG 只会把旧错误更稳定地送进模型。

6. **AI x Web3 层级设计**：
   - 指令层 → 任务层 → 事实层 → 知识层 → 记忆层
   - 层级越清楚，越容易做权限控制、prompt injection 防护和审计

### 与 Prompt 章节的联系

上一章学的 Prompt 是"如何构建好的指令"，这一章的 Context 是"模型能看到什么信息"。Prompt 和 Context 共同决定了模型输出的质量，缺一不可。

### 最小实践：钱包授权检查 Agent

章节末尾留了一个动手题——设计一份 context spec 分析"这个 dApp 要我 approve，可以签吗？"场景所需的上下文：
- chain id 和当前区块
- token 合约、spender 地址、approve 数量
- 用户当前 allowance 和余额
- spender 是否在可信列表
- simulation 或静态检查结果
- dApp 页面说明（标记为不可信）
- 用户本次意图

### 阅读：RAG（检索增强生成）

读了 [AI × Web3 School Handbook — RAG 章节](https://aiweb3.school/zh/handbook/ai/rag/)

### 关键概念 / 收获

1. **RAG 的本质** — 不是"给模型接一个向量库"，而是一条把外部知识取回、筛选、引用、交给模型使用的证据链。RAG 不是为了把回答变长，而是让回答有来源、有版本、有边界。

2. **缺少 citation 和 freshness 的 RAG** — 只是把幻觉从模型内部搬到了检索系统里。RAG 的可靠性取决于证据链，不取决于"向量库"这个名词。

3. **三个核心判断点**：
   - 文档怎么切（Chunking）
   - 查询时取哪些内容（Retrieval）
   - 生成时如何引用和拒答（Citation & Refusal）

4. **Chunking** — 按结构切（标题、API endpoint、函数说明、FAQ），每个 chunk 保留来源 URL、标题路径、更新时间和版本。技术文档要小心跨段落信息。

5. **Vector DB** — 不仅要存向量，还要存 metadata：来源、版本、chain、更新时间、可信等级、是否废弃。向量相似 ≠ 答案正确。

6. **Retriever** — 不能只看语义相似度，还要知道用户问的是哪个产品、版本、时间段。如果有版本/环境/时间/地址关键词，就不能只做纯向量搜索。

7. **Rerank** — 初筛后重新排序，把更可信的排前面。trade-off：增加延迟和成本，面向资产风险或开发者文档值得加。

8. **Citation 是 RAG 的灵魂** — 至少要说明：来自哪份文档、是否官方、版本/更新时间、哪些只是模型归纳、哪些地方没足够证据。

9. **RAG 在 AI x Web3 中的位置** — 位于 Knowledge Base 和模型之间，帮 Agent 查资料、补上下文、引用证据。但 RAG 结果不能直接触发链上动作，还要接 simulation、policy 和 human check。

### 与之前章节的联系

- **与 Context 章节**：RAG 本质上是在构建"外部知识层"的上下文。Context 章节提到的层级设计（指令→任务→事实→知识→记忆），RAG 负责的就是"知识层"那块。上一章说"模型看到了但没抓住重点"问题，RAG 用检索+rerank+citation 来解决。
- **与 Prompt 章节**：好的 RAG 输出结构需要好的 prompt 设计来约束——强制区分 answer / sources / uncertainties / needs_version_check。

### 最小实践

章节末尾的动手题：做一个"协议文档 RAG 问答"最小版本。选一个官方文档站，抓 10-20 篇页面，按标题结构切 chunk，存入向量库。用户提问时返回答案和引用。至少测试三类问题：明确存在的、不存在的（拒答）、旧版本的（提示核对版本）。

输出格式：
- answer
- sources
- uncertainties
- needs_version_check
